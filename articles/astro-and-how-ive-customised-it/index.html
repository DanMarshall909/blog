Building a cheap static website like this

25 June 2026

3 min read

  • Astro
  • static sites
  • frontend
  • developer tools

If you want a site like this without spending much money, a static website is a very good option. It is fast, cheap to host, and easy to maintain. For a personal site, blog, portfolio, or resume page, that is often all you need.

I built this blog with Astro because it gives me a solid foundation without making the setup feel heavy. But the bigger idea is simpler: you can get a polished, custom-looking website online for very little money.

What this kind of site looks like

A static site is just HTML, CSS, and maybe a little JavaScript. There is no backend, no database, and no server process to babysit. That means:

  • the pages are fast to load
  • the hosting cost is tiny
  • you can publish from GitHub
  • you can version the site in git

For a portfolio, resume, small business landing page, or personal blog, that is often more than enough.

The cheap stack

A simple setup can be:

  1. Astro for the site generator and content structure
  2. GitHub as the source of truth
  3. GitHub Pages for hosting
  4. a cheap domain name from a registrar like VentraIP

That is enough to get a modern-looking site online without paying for a VPS or a managed hosting plan.

Domain registration

If you want a proper web address, the domain registration step is usually the only recurring cost you need to think about. I use VentraIP for domain registration because it is straightforward and gives me the control I want over DNS records.

The workflow is simple:

  • buy the domain name
  • point it at GitHub Pages via the DNS settings
  • wait for the records to propagate
  • publish the site and it appears under your own domain

That is one of the nicest parts of this setup. You do not need to buy a big hosting package or learn a server control panel just to get a custom domain.

Hosting with GitHub Pages

GitHub Pages is a very practical hosting option for static sites. It is free for personal projects, it integrates well with GitHub repositories, and it works well with Astro builds.

The typical flow is:

  • push your site to a GitHub repo
  • enable GitHub Pages in the repo settings
  • tell it to publish from the build output
  • connect your custom domain

Once that is done, you have a site that is live and easy to update. For a blog or portfolio, that is hard to beat.

What this setup is good for

This approach is great when you want to publish:

  • a blog
  • a resume
  • a portfolio
  • a simple product landing page
  • a documentation site

It is not trying to be a full application platform. That is the point. It keeps the complexity low and the maintenance burden even lower.

Limitations

There are a few important limitations.

No database

If you need user accounts, comments, search over dynamic content, or a backend for storing data, a static site is not the right fit. You can still do a lot, but the data layer is usually external.

No VM

You also do not get a virtual machine to run arbitrary services. If you need background jobs, server-side rendering with custom runtime dependencies, or a full app stack, this approach becomes limiting.

No real-time features

Things like live chat, notifications, or custom APIs usually need a separate service. For a personal website, that is usually fine.

That said, for most static content needs, the trade-off is excellent. You get speed, simplicity, and low cost.

How I automate the boring parts

Once the site exists, the rest of the workflow becomes much easier. I have also written about how I automate things like keeping my resume current and publishing article updates to LinkedIn in my resume is now a static site feature and I built a /blogpost command that writes, illustrates, and publishes for me.

That is the real win here: the website is cheap to build, but the automation makes it feel much more like a system than a one-off project.

The short version

If you want a website that looks polished without costing much, a static site built with Astro, hosted on GitHub Pages, and served under a custom domain is a very strong option. It is simple enough to maintain, flexible enough to customise, and cheap enough that you can actually ship it.
